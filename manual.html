Angular 21 + NestJS + PostgreSQL — Flujo de trabajo en monorepo

💡

Este manual asume que ya completaste el setup inicial. Si es la primera vez que abres el proyecto, comienza por la sección Setup inicial.

📁 Estructura del proyecto

Cómo está organizado el monorepo y qué hace cada parte.

# Raíz del monorepo
portafolio/
├── apps/
│   ├── web/              # Frontend Angular 21
│   └── api/              # Backend NestJS 11
├── docker/
│   └── nginx/            # Reverse proxy (producción)
├── docs/                 # Documentación adicional
├── .env.example          # Variables de entorno (prod)
├── docker-compose.yml    # Stack completo (prod)
├── docker-compose.dev.yml # Solo PostgreSQL (dev)
└── package.json          # Scripts raíz del monorepo

¿Qué es un monorepo?

Es un único repositorio Git que contiene múltiples proyectos relacionados (web y api). Desde la raíz puedes ejecutar comandos que afectan a ambos. Cada workspace en apps/ tiene su propio package.json y puede ejecutarse de forma independiente.

✅

Todos los comandos de este manual se ejecutan desde la raíz del proyecto, salvo que se indique lo contrario.

⚙️ Setup inicial

Solo necesitas hacer esto una vez al clonar el repositorio.

1

Instalar dependencias

Instala todas las dependencias de la raíz y de ambos workspaces (web y api) con un solo comando:

npm install
2

Crear el archivo de entorno de la API

Copia el template y edítalo con tus valores locales:

cp apps/api/.env.example apps/api/.env

El archivo por defecto ya tiene valores para desarrollo local:

NODE_ENV=development
PORT=3000
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASS=postgres
DB_NAME=portafolio
JWT_SECRET=super-secret-change-me
JWT_EXPIRES_IN=7d
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=change-me
3

Levantar la base de datos con Docker

El entorno de desarrollo solo necesita PostgreSQL en Docker. El código corre directamente en tu máquina con hot-reload:

docker compose -f docker-compose.dev.yml up -d

# Verifica que el contenedor esté corriendo
docker compose -f docker-compose.dev.yml ps
🐳

Esto levanta solo PostgreSQL en el puerto 5432. La API y el frontend corren directamente en tu máquina, no en Docker.

4

Iniciar el entorno de desarrollo completo

Desde la raíz, este comando inicia la API y el frontend en paralelo con hot-reload:

npm run dev

# O por separado si prefieres ver los logs divididos:
npm run api   # Terminal 1 → NestJS en :3000
npm run web   # Terminal 2 → Angular en :4200
5

Verificar que todo funciona

Abre estas URLs en tu navegador:

Frontend
http://localhost:4200
API Health Check
http://localhost:3000/api

La API también crea automáticamente el usuario admin al iniciar (usando ADMIN_EMAIL y ADMIN_PASSWORD del .env).

🟦 Cambios en la API (NestJS)

API Todo lo relacionado con apps/api/

1

Asegúrate de que el entorno esté corriendo

# PostgreSQL en Docker
docker compose -f docker-compose.dev.yml up -d

# API con hot-reload (desde raíz)
npm run api

La API se recargará automáticamente cada vez que guardes un archivo en apps/api/src/.

2

Estructura de un módulo NestJS

Para agregar funcionalidad nueva, sigue este patrón. Por ejemplo, un módulo contact:

apps/api/src/modules/contact/
├── contact.module.ts      # Declara el módulo
├── contact.controller.ts  # Define los endpoints (rutas)
├── contact.service.ts     # Lógica de negocio
├── dto/
│   └── create-contact.dto.ts  # Validación de datos entrantes
├── entities/
│   └── contact.entity.ts  # Modelo de base de datos
└── contact.spec.ts        # Tests unitarios

Luego importas el módulo en app.module.ts.

3

Probar los endpoints manualmente

Usa curl, Postman, o Insomnia. Primero obtén el JWT:

# 1. Login para obtener el token
curl -X POST http://localhost:3000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"change-me"}'

# 2. Usar el token en rutas protegidas
curl http://localhost:3000/api/projects \
  -H "Authorization: Bearer TU_TOKEN_AQUI"

# Endpoints públicos (sin token)
curl http://localhost:3000/api/projects   # GET proyectos
curl http://localhost:3000/api/newsletter  # GET newsletter
4

Ejecutar los tests

# Tests unitarios (desde raíz o desde apps/api)
npm run test --workspace=apps/api

# Con watch mode (se re-ejecutan al guardar)
cd apps/api && npm run test:watch

# Con reporte de cobertura
cd apps/api && npm run test:cov

# Tests E2E (necesita la DB corriendo)
cd apps/api && npm run test:e2e
5

Verificar tipos y lint

# Lint solo la API
cd apps/api && npm run lint

# O desde la raíz (lint ambos)
npm run lint

Ejemplo completo — agregar un nuevo endpoint

📝

Supongamos que quieres agregar GET /api/skills para listar habilidades técnicas.

Paso A — Crear la entidad

// apps/api/src/modules/skills/entities/skill.entity.ts
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

@Entity()
export class Skill {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  @Column()
  level: number;  // 1-100
}

Paso B — Crear el módulo

// apps/api/src/modules/skills/skills.module.ts
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { Skill } from './entities/skill.entity';
import { SkillsController } from './skills.controller';
import { SkillsService } from './skills.service';

@Module({
  imports: [TypeOrmModule.forFeature([Skill])],
  controllers: [SkillsController],
  providers: [SkillsService],
})
export class SkillsModule {}

Paso C — Registrar en app.module.ts

// apps/api/src/app.module.ts — agrega SkillsModule al array imports
imports: [
  // ... otros módulos existentes
  SkillsModule,
],

La API se recargará automáticamente y TypeORM creará la tabla skill en PostgreSQL (porque synchronize: true está activo en desarrollo).

⚠️

synchronize: true solo es seguro en desarrollo. En producción se usan migraciones para no perder datos.

WEB Todo lo relacionado con apps/web/

Iniciar el servidor de desarrollo

# O directamente dentro del workspace

Angular recompila y recarga el navegador automáticamente al guardar cualquier archivo.

2

Estructura del frontend

apps/web/src/app/
├── core/
│   ├── guards/       # AuthGuard (rutas protegidas)

│   └── services/     # AuthService, etc.
├── features/

│   └── admin/        # Dashboard, CRUD (requiere login)
└── shared/           # Componentes reutilizables
3

Crear un componente nuevo

Usa Angular CLI desde dentro del workspace:

cd apps/web

# Componente público (standalone)
npx ng generate component features/public/skills --standalone

# Componente en admin
npx ng generate component features/admin/skills-manager --standalone

# Servicio para consumir la API
npx ng generate service core/services/skills
4

Consumir la API desde Angular

El HttpInterceptor ya agrega el JWT automáticamente a las peticiones. Solo necesitas inyectar HttpClient:

// apps/web/src/app/core/services/skills.service.ts
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { environment } from '../../environments/environment';

@Injectable({ providedIn: 'root' })
export class SkillsService {
  private url = `${environment.apiUrl}/skills`;

  constructor(private http: HttpClient) {}

  getAll() {
    return this.http.get(this.url);
  }
}
5

Agregar una ruta nueva

// apps/web/src/app/app.routes.ts
export const routes: Routes = [
  // Ruta pública
  {
    path: 'skills',
    loadComponent: () =>
      import('./features/public/skills/skills')
        .then(m => m.SkillsComponent)
  },
  // Ruta protegida (admin)
  {
    path: 'admin/skills',
    canActivate: [AuthGuard],
    loadComponent: () =>
      import('./features/admin/skills-manager/skills-manager')
        .then(m => m.SkillsManagerComponent)
  },
];

🧪 Tests de la API — Jest

TEST Framework: Jest. Archivos: *.spec.ts dentro de apps/api/src/

Tipos de tests disponibles

Tests unitarios

Prueban servicios y lógica de negocio de forma aislada (sin BD real).

cd apps/api && npm test

# Solo un archivo específico
npm test -- --testPathPattern=projects.service

# Watch mode
npm run test:watch

Ejemplo de test unitario

// apps/api/src/modules/skills/skills.service.spec.ts
describe('SkillsService', () => {
  let service: SkillsService;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        SkillsService,
        { provide: getRepositoryToken(Skill), useValue: mockRepo },
      ],
    }).compile();

    service = module.get(SkillsService);
  });

  it('should return all skills', async () => {
    const result = await service.findAll();
    expect(result).toHaveLength(2);
  });
});

Requieren que PostgreSQL esté corriendo: docker compose -f docker-compose.dev.yml up -d

🐘 CloudBeaver — UI para la base de datos

Cliente web para ejecutar queries SQL directamente contra PostgreSQL, incluido en el docker-compose.yml.

Cómo acceder

CloudBeaver corre como un servicio más del stack. En Coolify debes exponer el puerto 8978 asignándole un dominio o subdominio.

CloudBeaver (producción)
tu-subdominio.miguelangeljaimen.cl
Puerto interno
8978

Configurar la conexión a PostgreSQL

La primera vez que abres CloudBeaver te pedirá crear una conexión. Usa estos valores — todos los servicios están en la misma red Docker del compose:

Driver:    PostgreSQL
Host:      db             # nombre del servicio en docker-compose.yml
Port:      5432
Database:  portafolio
User:      postgres
Password:  <valor de DB_PASS>
✅

El host es db (no localhost) porque CloudBeaver y PostgreSQL comparten la red interna de Docker Compose y se resuelven por nombre de servicio.

Configurar en Coolify

1

Exponer el puerto 8978

En Coolify → tu servicio → Settings del contenedor cloudbeaver → agrega un dominio apuntando al puerto 8978.

2

Redeploy

Después de guardar, haz Deploy para que Coolify recree el contenedor con la nueva configuración de red.

3

Primera vez — crear admin

CloudBeaver te pedirá crear un usuario administrador local en el primer acceso. Este usuario es solo para la UI de CloudBeaver, no tiene relación con el admin de la API.

⚠️

CloudBeaver tiene acceso completo a la base de datos. Asegúrate de proteger el subdominio con una contraseña fuerte o restringirlo por IP si es posible.
